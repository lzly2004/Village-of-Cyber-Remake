    新一轮代码全面扫描 — 发现的重构方向（2026-06-29）
================================================================================

经过对项目 ~85个Java文件、12,000+行代码的全面扫描和深度分析，
识别出以下 8 个有价值的重构方向，按优先级排序：

================================================================================
🔥 高优先级（架构意义 / 质量提升显著）
================================================================================

【P3-1】ExecutionManager.execute() 职责拆分
  工作量: 中等（4-6小时）
  风险: 低
  收益: 高
  综合优先级: ⭐⭐⭐⭐
  现状:
    - execute() 方法约 200 行，包含多个阶段：
      1. 预告日志输出（~70行）
      2. 回避CO处理
      3. 投票资格判定
      4. 多轮投票执行
      5. 投票数据录制
      6. 处刑与猫咒处理
      7. 游戏结束判定与战绩更新
      8. 进入下一夜
    - 方法内嵌套深，可读性较差
  方案:
    - 拆分为 5-7 个语义化子方法
    - 保持 execute() 作为协调者
    - 参考 NonHumanCoordinator.coordinate() 的拆分模式
  风险点:
    - 需确保状态变更顺序不变
    - 需完整测试投票流程
  建议: 优先实施，与 P1-10/P1-10.2 形成完整的长方法拆分系列

【P3-2】SuspicionSystem.updateTop3SuspectedPlayers() 拆分
  工作量: 中等（3-5小时）
  风险: 低
  收益: 高
  综合优先级: ⭐⭐⭐⭐
  现状:
    - 方法约 110 行，三重嵌套循环
    - 包含多个逻辑阶段：
      1. 基础怀疑度随机波动（按阵营不同概率）
      2. CO角色怀疑度修正（占/灵/猎）
      3. 懒怀疑度应用
      4. 怀疑度值收束（MAXN/INF边界）
      5. 占CO白球全局加权
      6. 懒怀疑度清零
      7. Top3怀疑玩家计算
    - 嵌套层级深（4层for+switch+if）
  方案:
    - 拆分为 4-5 个子方法
    - 每个方法负责一个逻辑阶段
    - 提高可读性和可测试性
  建议: 优先实施，核心逻辑方法应保持清晰

【P3-3】MainLogic 中 GameStatus 直接访问封装（Phase 3 收尾）
  工作量: 小（1-2小时）
  风险: 低-中
  收益: 中-高
  综合优先级: ⭐⭐⭐⭐
  现状:
    - MainLogic 中有 ~25 处直接访问 gs.xxx
    - 包括: gs.startGame(), gs.gameDay++, gs.end, gs.p, gs.gc[i]
    - 与 GameContext 的封装目标不一致
    - 破坏分层架构（Logic 层应通过 Context 访问状态）
  涉及字段:
    - gs.gameDay → ctx.getGameDay() / ctx.incrementGameDay()
    - gs.end → ctx.getEndResult() / ctx.setEndResult()
    - gs.p → ctx.getPeiyi()
    - gs.gc[i] → 已有 ctx 方法
    - gs.getPlayerSum() → ctx.getPlayerSum()
  方案:
    - GameContext 补充缺失的方法（incrementGameDay等）
    - MainLogic 中所有 gs. 替换为 ctx.
    - gs 字段改为 private，仅用于初始化
  收益:
    - 完成 GameContext 作为唯一状态访问入口的架构目标
    - 统一状态访问路径，便于后续维护
  建议: 优先实施，架构一致性改进

================================================================================
⚡ 中优先级（代码质量 / 可维护性提升）
================================================================================

【P3-4】游戏结果枚举化（GameResult 枚举）
  工作量: 小（1-2小时）
  风险: 低
  收益: 中
  综合优先级: ⭐⭐⭐
  现状:
    - GameStatus.end 用 int 表示: 0=未结束, 1=村胜, 2=狼胜, 3=狐胜
    - 分布在 ~10 个文件中，约 20 处比较
    - 类型不安全，容易写错数字
  方案:
    - 新增 GameResult 枚举: NONE, VILLAGER_WIN, WOLF_WIN, FOX_WIN
    - GameStatus.end 类型改为 GameResult
    - 替换所有数字比较为枚举比较
  涉及文件:
    - GameStatus, GameContext, MainLogic, ExecutionManager
    - GameEndChecker, GameRecord, ReplaySave
    - EndGameHandler, EndAnimeHandler 等
  建议: 中等优先级，小而美的类型安全改进

【P3-5】5个UI工厂类合并为 UIFactory
  工作量: 小（2-3小时）
  风险: 低
  收益: 中
  综合优先级: ⭐⭐⭐
  现状:
    - 5个 SimpleFactory 类分散:
      ButtonSimpleFactory, LabelSimpleFactory, PanelSimpleFactory
      TextareaSimpleFactory, (可能还有其他)
    - 每个类只有几个静态方法
    - 类数量多但功能相关
  方案:
    - 合并为一个 UIFactory 类
    - 按组件类型分组静态方法
    - 保留原方法签名，调用方只需改 import
  涉及文件:
    - 5个工厂类 + 25个调用类
  建议: 中等优先级，减少类数量，统一管理

【P3-6】WolfBiter.decideBite() Phase 1 派生值计算拆分
  工作量: 小-中（2-3小时）
  风险: 低
  收益: 中
  综合优先级: ⭐⭐⭐
  现状:
    - decideBite() 方法约 320 行
    - Phase 1（派生值计算）约 140 行
    - 包含: 末狼判定、初日占被咬判定、真占/真灵/真猎识别、
      白球计数、猎人护卫系数计算、白球全局计数等
    - 变量多（y, z, zhenZhanCnt, baiqiu, k, hiddenHunterCnt...）
  方案:
    - Phase 1 提取为独立方法或内部类
    - 派生值封装为一个 Context 对象
    - 提高 Phase 2/3/4 的可读性
  建议: 中等优先级，WolfBiter 是核心AI逻辑
