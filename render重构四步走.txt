### 核心问题：`render()` 方法的930行困境

这个方法是典型的**巨型方法（God Method）**，它违反了以下原则：

#### 1. 单一职责原则（SRP）—— 最严重的违反

`render()` 同时做了 **7件完全不同的事**：

| 职责 | 行数 | 应归属的组件 |
|------|------|------------|
| 布置BGM | ~25行 | `BgmSelector` |
| 渲染玩家状态 | ~92行 | `PlayerStatusRenderer` |
| 渲染信息面板 | ~112行 | `VoteInfoRenderer` |
| 创建按钮面板 | ~82行 | `VoteButtonPanel` |
| 计算灰名单 | ~29行 | `GreyListCalculator` |
| 投票/CO/指定事件处理 | ~350行 | `VoteEventDispatcher` |
| CO选择面板（含嵌套） | ~240行 | `CoSelectionPanel` |

#### 2. 开闭原则（OCP）—— 添加新按钮需要修改930行的方法

如果要添加一个新按钮（比如"遗言"按钮），开发者必须在 ~930 行的代码中找到合适的位置插入，同时确保 `returnBtn` 的 `setVisible` 列表里也加上新按钮的隐藏逻辑。当前 `returnBtn` 的 lambda 里就有 **23行** 的 `setVisible(false)` 调用：

```java
// L640-L660: returnBtn 的 ActionListener
voteBtn.setVisible(true); pointBtn.setVisible(true); infoText.setVisible(true);
ui.piaoText.setVisible(false); ui.piaoText1.setVisible(false);
// ... 20+ 行 setVisible 调用
votehisBtn.setVisible(false); doubtBtn.setVisible(false); recordBtn.setVisible(true);
coBtn.setVisible(false); ppBtn.setVisible(false); returnBtn.setVisible(false);
reiBtn.setVisible(false); kariBtn.setVisible(false); uranaiBtn.setVisible(false);
kyouyuBtn.setVisible(false); catBtn.setVisible(false); askCoBtn.setVisible(false);
greyBtn.setVisible(false); freeBtn.setVisible(false); readyVoteBtn.setVisible(false);
nextBtn.setVisible(false); againBtn.setVisible(false);
```

这 23 行本质上是**状态机的手动实现**。每次添加新按钮，开发者必须记住在所有这些位置同步更新。

#### 3. 依赖倒置原则（DIP）—— 直接依赖 `ui.gs.gc[i]` 的具体实现

`render()` 中有 **290+ 处 `ui.gs` 直接访问**，这意味着：
- 任何对 `GameStatus` 内部结构的修改都可能破坏这里的渲染逻辑
- 无法为 UI 逻辑编写单元测试（因为必须启动完整的 `GameStatus`）
- 无法替换 `GameStatus` 为其他实现

### 重构方案：分阶段演进

考虑到Swing UI的固有复杂性（组件间高度耦合），建议采用**三步走**策略，而不是一次性大手术：

#### 阶段一：Extract Method（零风险，立即可做）

将 `render()` 拆分为命名的私有方法序列，不改变任何行为：

```java
public void render(UI ui) {
    this.ui = ui;
    ui.jPanel.removeAll();
    renderBackground(ui);
    renderMusicButton(ui);
    renderPlayerStatuses(ui);       // L42-L133
    renderSkillTargets(ui.jPanel, 160, 64, 0, 98);
    VoteInfoPanel info = renderVoteInfo(ui);      // L136-L247
    VoteButtons buttons = createButtonPanel(ui);  // L248-L330
    List<Integer> greyList = computeGreyList(ui); // L331-L359
    wireVoteHandlers(ui, info, buttons, greyList); // L360-L595
    wireSpecifyHandlers(ui, buttons);              // L596-L629
    wireCoSelectionHandlers(ui, buttons);          // L630-L943
}
```

**收益**：`render()` 从 930 行 → ~30 行。每个子方法职责单一、可独立理解。这是**完全安全的**——不改变任何逻辑，只是代码组织。

**行数变化**：`render()` -900 行，新增 8 个私有方法 +900 行，净变化 **0**。

#### 阶段二：Extract Class（中等风险，需要测试）

将阶段一提取的方法进一步迁移到独立的类中：

```
GameSceneVoteHandler (协调器，~80行)
├── VoteInfoRenderer      (~150行)  // 信息面板渲染
├── VoteButtonPanel       (~200行)  // 按钮创建 + 可见性管理
├── GreyListCalculator    (~40行)   // 灰名单计算（纯逻辑，可单测）
├── VoteEventDispatcher   (~200行)  // 投票/CO/指定事件分发
└── CoSelectionPanel      (~250行)  // CO选择面板
```

**关键设计**：`VoteButtonPanel` 封装按钮可见性管理，用 `Map<String, JButton>` 替代散落的23个局部变量，并用 `setMode(Mode)` 方法统一管理可见性切换：

```java
// 替代那23行 setVisible，变成：
class VoteButtonPanel {
    enum Mode { DEFAULT, VOTING, CO_SELECTING, SPECIFYING }
    
    void setMode(Mode mode) {
        buttonMap.forEach((name, btn) -> 
            btn.setVisible(mode.isVisible(name))
        );
    }
}
```

**收益**：
- 每个类可独立测试（`GreyListCalculator` 甚至不需要 UI）
- 添加新按钮只需在 `VoteButtonPanel` 中注册，无需修改其他文件
- `CoSelectionPanel` 中的3层嵌套匿名内部类（L800-L943）有了清晰的边界

#### 阶段三：引入 GameContext 替代 `ui.gs`（高收益，但依赖发现一完成）

将 `ui.gs.gc[i].whyDie`、`ui.gs.gameDay`、`ui.gs.aliveCounter` 等 290+ 处直接访问替换为 `ctx.getDeathReason(i)`、`ctx.getGameDay()` 等。这需要 `GameContext` 已经提供了足够的方法。
